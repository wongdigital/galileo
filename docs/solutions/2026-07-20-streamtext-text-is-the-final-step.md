# `streamText().text` is the final step, not the answer

## The symptom

The chat bubble rewrote itself. The model would stream "Let me check the Hall H panels…",
call a tool, come back with the answer—and the moment the turn finalized, the prose the user
had just watched appear was gone, replaced by whatever the last step happened to say.

A second symptom looked unrelated but shares a cause. Pressing **Stop** mid-turn, or letting
the 90-second timeout fire, sometimes returned `ok: true` with a half-answer instead of
"Stopped." And because the success path commits `capture.patch`, an aborted turn could still
commit a filter change to app state—the user cancels, and the sidebar moves anyway.

Both were found by review, not by use. Neither is visible in a single-step turn, which is
what most manual testing produces.

## Why it happens

Two promise semantics in AI SDK v7 (`ai@7.0.31`) that read as safe and are not.

**`.text` is scoped to one step.** The JSDoc in `node_modules/ai/dist/index.d.ts` says it
plainly: *"The full text that has been generated by the final step."* Four lines up, the
adjacent `content` field says *"The content that was generated in all steps."* So `.text` is
a per-step accessor wearing a whole-result name. Any turn where the model speaks, calls a
tool, then speaks again loses the first half.

**An abort after a completed step resolves rather than rejects.** In
`node_modules/ai/dist/index.js`, the recording transform's `flush` rejects the result
promises only when `recordedSteps.length === 0`. With one or more steps recorded it falls
through and resolves normally, with partial data. The complementary half: when the abort
signal is seen during `pull`, the SDK enqueues an `{ type: "abort" }` part and calls
`controller.close()`—a graceful close, not `controller.error()`. Even the catch arm routes
abort errors back into that same graceful path.

So abort-before-any-step rejects, and abort-after-a-step resolves. The original code handled
aborts only in `catch`, which covers exactly one of those two cases—and the rarer one.

Line numbers shift between patch versions. Re-verify on your own install by searching
`index.d.ts` for `generated by the final step` and `index.js` for `recordedSteps.length === 0`.

## The fix

**Accumulate the stream you are already iterating.** The loop was walking `fullStream` purely
to forward deltas over IPC and throwing the text away. Keeping it costs nothing and is the
only source of what the user actually saw.

```ts
let streamed = ''
for await (const part of first.fullStream) {
  if (part.type === 'text-delta') {
    streamed += part.text
    onDelta?.({ text: part.text })
  } else if (part.type === 'tool-call') {
    onDelta?.({ status: TOOL_STATUS[part.toolName] ?? 'Working…' })
  }
}
const finalText = await first.text
if (finalText.trim()) return { text: streamed.trim() ? streamed : finalText }
```

**Check the abort signal on the resolve path too**, via an extracted `abortResponse(signal)`
so both paths discriminate stop-vs-timeout identically:

```ts
const { text } = await generate({ ... })
// An abort after ≥1 completed step resolves rather than rejecting, so a Stop
// or timeout can land here with a half-answer.
if (deps.signal?.aborted) return abortResponse(deps.signal)
return { ok: true, turn: { ... } }
} catch (error) {
  if (deps.signal?.aborted) return abortResponse(deps.signal)
  return { ok: false, error: classifyError(error) }
}
```

The check sits *before* the success path's side effects, so a cancelled turn commits nothing.

**A third defect surfaced in the same function.** The empty-reply fallback summarizer called
`streamText` without `tools`, with a comment reading "tools withheld"—a correct intent
implemented with the wrong lever. Once the replayed history contains `tool_use`/`tool_result`
parts, the provider requires the schema to ride along. Withholding the *schema* is not how
you withhold *use*:

```ts
const summary = streamText({
  model, system,
  messages: [...messages, ...gathered.messages],
  tools,               // history replays tool_use/tool_result—schema must ride along
  toolChoice: 'none',  // …but the model may not call anything
  abortSignal: signal,
})
```

The separator between streamed prose and the forced summary is both appended to `streamed`
and emitted as a delta, so the accumulated string and the on-screen text stay byte-identical.

## Prevention

`src/main/llm/__tests__/loop.stream.test.ts` mocks the `ai` module with a scripted
`fakeStream(parts, finalText, responseMessages)`, so every branch is observable without a
provider or a key. The shape worth copying is that `parts` and `finalText` are allowed to
**disagree**—parts say `'Hello '` then `'world'`, `.text` says only `'world'`. A fake where
they always agree cannot catch this class of bug at all.

```ts
// two steps wrote prose; the final step's text is only 'world'
if (res.ok) expect(res.turn.message.content).toBe('Hello world')

// abort on the RESOLVE path: the stream yields text and closes cleanly
streamText.mockReturnValueOnce(fakeStream([{ type: 'text-delta', text: 'Half' }], 'Half'))
controller.abort(new Error('cancelled'))
expect(res.ok).toBe(false)
if (!res.ok) expect(res.error.kind).toBe('aborted')

// the summarizer contract, pinned on call args rather than output
const summaryArgs = streamText.mock.calls[1]![0] as { tools?: unknown; toolChoice?: unknown }
expect(summaryArgs.tools).toBeDefined()
expect(summaryArgs.toolChoice).toBe('none')
```

What generalizes past this SDK:

- When a streaming library exposes both an iterable and a convenience promise over the same
  data, read the promise's doc comment for scope words—*final*, *last*, *current*—before
  treating it as the whole result. Prefer accumulating from the iterable you already consume.
- Abort handling belongs on **both** the resolve and the reject path of a multi-step
  streaming call. The catch-only version is invisible to any test that makes the stream
  throw; the test that catches it is one where an already-aborted signal resolves normally.
- Treat *aborted* as a third outcome, distinct from success and error, and put the success
  path's side effects behind the abort check rather than in front of it.
